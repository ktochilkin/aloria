# CLAUDE.md — инструкции ассистенту по работе с Aloria

> Этот файл автоматически подгружается Claude Code в каждой сессии. Сюда попадает **источник правды** про продукт и про то, как с ним работать. Если что-то в проекте противоречит этому файлу — приоритет за ним. Если хочется поменять правило — сначала обсудить с пользователем, потом обновить этот файл.

## 1. Что такое Aloria

Aloria — **учебный симулятор фондового рынка** (Flutter, iOS / Android / Web) с реальной торговой логикой:

- настоящий мэтчинг заявок,
- реальные типы заявок,
- рыночное поведение цены,
- но **без реальных денег**.

Это безопасная среда, чтобы понять, как работает рынок и риск, с игровыми механиками для вовлечения и удержания.

**Философия продукта** (важно для UX и текстов):

- Пользователь сначала **понимает**, а потом получает больше возможностей.
- Внутренняя валюта — это **мера допуска и ответственности** и одновременно измеримый результат прогресса.
- Ошибки — нормальная часть обучения.
- Игровые механики (уровни, опыт, прогресс, стрики, ачивки, челленджи, лидерборды, ежедневные цели) допустимы и приветствуются — они работают на удержание и обучение.
- Геймификация живёт **поверх** торгового интерфейса, а не вместо него: типы заявок, стакан, поведение цены остаются достоверными. Не превращаем рынок в игровую механику — геймифицируем путь обучения и вовлечения.

### 1.1. Бизнес-контекст

- Aloria — **некоммерческий** энтузиаст-проект автора **внутри брокера Алор**.
- Цель — бесплатное, качественное, интересное обучение фондовому рынку **в игровой форме**.
- Если продукт станет популярным — это улучшение бренда Алор, не источник выручки.
- **Никаких пейволлов, премиум-уровней, платных курсов, платных бейджей**. Весь контент и весь функционал — бесплатно для пользователя.
- Никакой агрессивной монетизации, рекламы, продажи лидов, ретаргетинга.
- Из этого следует: оптимизируем продукт **под качество обучения и удовольствие**, а не под конверсии и LTV. Это влияет на дизайн механик — например, мы не будем намеренно «фрустрировать чтобы продать», как это делают free-to-play игры.

## 2. Стек и структура

- **Flutter** (stable, SDK `>=3.10.0 <4.0.0`)
- **State**: `flutter_riverpod` (`Provider`, `FutureProvider`, `StateNotifierProvider`)
- **Routing**: `go_router` с `StatefulShellRoute.indexedStack` (нижняя навигация из 3 разделов)
- **Networking**: `dio` + типизированные ошибки (`core/errors`)
- **Realtime**: `web_socket_channel` (`core/networking/realtime_client.dart`)
- **Models**: `freezed` + `json_serializable`
- **Storage**: абстракция `Storage` (secure + prefs на мобилке, prefs на вебе)
- **Theming**: Material 3 + токены (`core/theme/tokens.dart`, `app_theme.dart`)
- **Logger**: `logger` (через `core/logging/logger.dart`)
- **Markdown**: `flutter_markdown` (для контента обучения)

Команды:
- `flutter pub get`
- `dart run build_runner build --delete-conflicting-outputs` — после правок freezed/json моделей
- `flutter analyze` — обязательно перед PR
- `flutter test`
- Запуск:
  - iOS: `flutter run --release -d <device_id> --dart-define=APP_ENV=dev --dart-define=API_BASE_URL=... --dart-define=WS_BASE_URL=wss://api.alor.dev/ws --dart-define=AUTH_BASE_URL=... --dart-define=ENABLE_LOGGING=true`
  - Android: `flutter run --release -d android --dart-define=...`
  - Web (Chrome): `flutter run -d chrome --dart-define=...`

## 3. Кодовая дисциплина (главные правила)

> Эти правила обязательны. Если запрос пользователя их нарушает — **обсудить альтернативу**, не молча идти на компромисс.

### 3.1. Без костылей

- Решения должны быть **стандартными** для Flutter. Никаких хаков «лишь бы заработало», никаких `if Platform.isIOS` ради того, чтобы скрыть проблему вместо её решения.
- Если знаешь, что выбранный путь — компромисс или костыль (workaround вокруг бага платформы, специфичный API, нестабильный пакет, и т.п.) — **сообщи об этом пользователю до того, как писать код**, и предложи альтернативы.
- Если у пользователя запрос, который реализуется только костылём — обсуди, **что именно мешает** сделать чисто, и предложи варианты (упростить требование, использовать другой подход, поднять с бэка / другой источник данных).

### 3.2. Кросс-платформенность

- Поддерживаемые платформы: **iOS, Android, Web (Chrome)**. Все изменения должны работать на всех трёх.
- Если функциональность принципиально не работает на одной из платформ (например, аппаратный API недоступен в браузере) — обсудить заранее.
- Опасные места, где обычно ломается кросс-платформенность:
  - `dart:io` (нет на вебе) → закрывать через абстракцию (см. `core/storage/storage_factory.dart` + `_io.dart` / `_web.dart`).
  - Файловая система, push, биометрия, Bluetooth — отдельные пакеты с web-фолбэками.
  - WebSocket: на вебе ходит через `HtmlWebSocketChannel`, на мобилке — `IOWebSocketChannel`. `web_socket_channel` это абстрагирует, не лезть напрямую в `dart:io.WebSocket`.
  - Фокус, клавиатура, скролл-физика на вебе ведут себя иначе. Использовать `core/platform/web_scroll_behavior.dart` для прокрутки.
  - Фонтбэки (`google_fonts`) на мобилке могут грузиться с задержкой при первом запуске — учитывать в UI.
- Перед PR:
  - сборка хотя бы трогается на одной мобильной (`flutter build ios --no-codesign` / `flutter build apk --debug`),
  - и на вебе (`flutter build web`).

### 3.3. Сторонние пакеты — можно, но осторожно

- Сторонние пакеты допустимы, если это даёт **существенно лучшее** решение, чем писать руками.
- Перед добавлением зависимости проверь: поддерживает ли пакет все три платформы, давно ли обновлялся, есть ли активные issue про iOS/Web, размер бандла, лицензия.
- Если пакет добавляется ради одной мелочи — обсудить, может проще написать самому.
- В `pubspec.yaml` фиксировать версии в формате caret (`^x.y.z`).

### 3.4. Архитектура

Проект **feature-first**, слои внутри фичи:

```
lib/features/<feature>/
├── data/                 # репозитории, источники данных, DTO
├── domain/               # модели, бизнес-логика, use cases
├── application/          # Riverpod-провайдеры, контроллеры/notifier'ы
└── presentation/         # экраны и UI-виджеты фичи
```

Правила:

- **Не раздувать классы.** Если виджет / класс / файл становится больше ~400 строк — это сигнал делить. Большие экраны разбивать на подвиджеты в том же файле, или в отдельный файл если виджет переиспользуется. См. как сделано в `features/learn/presentation/`.
- **Тонкий UI.** В виджетах — только композиция и работа с состоянием через провайдеры. Бизнес-логика — в `domain/` или в notifier'ах.
- **Repositories** инкапсулируют сеть и хранилище, отдают доменные модели.
- **Riverpod**: `Provider` для синглтонов и derived-данных, `FutureProvider` для ленивых ассинхронных вычислений, `StateNotifierProvider` для наблюдаемого состояния. Не использовать глобальные `ChangeNotifier` напрямую без Riverpod.
- **Никаких side-effects в `build`.** Перенос в `initState`, `Future.microtask`, `ref.listen`, или в notifier.
- **Freezed-модели** для сложных value-объектов (несколько полей, equality, copyWith). Для простых структур (3 поля, без equality) — обычный класс с `const` конструктором.
- **Generated-файлы** (`*.g.dart`, `*.freezed.dart`) **не коммитятся вручную** — только результат `build_runner build --delete-conflicting-outputs`. Не править их руками.

### 3.5. Стиль кода

- `analysis_options.yaml` подключает пакет `lint`. Дополнительно включён `prefer_single_quotes` и `strict-raw-types`. **`flutter analyze` должен быть зелёным.**
- Одинарные кавычки везде (`'строка'`).
- Импорты: `package:` абсолютные, а не относительные.
- Имена: классы — `UpperCamelCase`, файлы — `lower_snake_case.dart`, приватные начинаются с `_`.
- Public API классов / методов комментируется на русском, в дартдок-стиле (`///`).
- Никаких `print` в продовом коде — только через `logger`.

### 3.6. Process: что обсуждать ДО кода

Если запрос пользователя приводит к одной из ситуаций ниже — **сначала вынести на обсуждение**, потом писать:

- Решение работает только на одной платформе.
- Нужно добавить новую зависимость в `pubspec.yaml`.
- Нужно изменить публичные API существующих провайдеров / репозиториев / моделей (потенциальный breaking change).
- Один файл/класс становится больше ~400 строк.
- Нужно поменять архитектурные слои (например, тащить сетевые вызовы в виджет).
- Нужен `Platform.isIOS` / `kIsWeb` ветка в логике (а не в инфраструктуре).
- Нужно добавить миграцию данных, ломающую сохранённое состояние пользователя.

Формат: 1–2 предложения «вижу такую сложность, варианты А / Б / В, рекомендую Б, продолжать?».

## 4. Структура приложения

### 4.1. Маршрутизация (`lib/router.dart`)

Нижняя навигация — `StatefulShellRoute.indexedStack` с тремя ветками:

| Раздел       | Маршрут        | Иконка          |
|--------------|----------------|-----------------|
| Обучение     | `/learn`       | `school`        |
| Портфель     | `/positions`   | `list_alt`      |
| Обзор рынка  | `/market`      | `show_chart`    |

Подмаршруты:
- `/learn/:sectionId` — список уроков раздела
- `/learn/:sectionId/:lessonId` — экран урока
- `/market/:symbol` — карточка инструмента / TradePage

Логин — `/login`, доступ к остальным маршрутам — после авторизации (refreshListenable на `authControllerProvider.stream`).

### 4.2. Фичи

- `features/auth` — логин, рефреш токена, хранение в `Storage`.
- `features/learn` — обучение (разделы, уроки, прогресс, мини-тесты). Контент — markdown в `assets/lessons/`. Прогресс — SharedPreferences.
- `features/market` — рынок: список инструментов, торговая страница (карточка инструмента, лента сделок, стакан, свечи, форма заявки), позиции и заявки.
- `features/example` — эталонная вертикальная нарезка (repository → use case → provider → UI). Брать как образец.

### 4.3. Реальное состояние торговли

- Биржа: **`TEREX`** (захардкожено).
- Портфель по умолчанию: **`T00013`**.
- WebSocket-подписки (`features/market`):
  - `QuotesSubscribe`
  - `OrderBookGetAndSubscribe` (depth=10, frequency=250 ms)
  - `BarsGetAndSubscribe`
  - `PositionsGetAndSubscribeV2`
  - `SummariesGetAndSubscribeV2`
  - `OrdersGetAndSubscribeV2`

### 4.4. Конфигурация (`core/env/env.dart`, `core/env/app_config.dart`)

- `AppEnv`: `dev`, `stage`, `prod`.
- `AppConfig`: `apiBaseUrl`, `wsBaseUrl`, `authBaseUrl`, `enableLogging`.
- Передаётся через `--dart-define`. Никаких хардкоженных URL'ов.

## 6. Тон и формулировки в UI

### 6.1. Допустимые формулировки про доступ, прогресс и валюту

- «теперь доступно», «разрешено использовать», «открывается возможность»
- «пройдено», «доступно», «сейчас» (про прогресс обучения)
- «пройдено / в процессе / не начат» (статусы разделов)
- «уровень», «опыт» / «XP», «прогресс», «достижение», «ачивка», «бейдж»
- «стрик», «серия», «ежедневная цель», «челлендж», «миссия», «квест»
- «награда», «бонус», «получено», «разблокировано» — для игровых механик (XP, бейджи, разблокировка контента, расширение покупательной способности по итогам обучения)
- Лидерборды, рейтинги, сезоны — допустимо.

### 6.2. Запрещено

- Преувеличение / упрощение риска (даже в игровой обёртке — «ставка», «крутани и узнай», «лёгкие деньги» и т.п.).
- Персональные финансовые / инвестиционные советы («покупай X», «продавай Y»).
- Обещания доходности.
- Игровая лексика, размывающая природу торговли: открытие позиции — это **не** «ставка», убыток — **не** «штраф», рынок — **не** «казино», лимитная заявка — **не** «загадай число». Геймифицируем процесс обучения и вовлечения, а не саму суть рынка.
- Гэмблинговые механики: лутбоксы, RNG-награды за сделки, бесплатные «прокрутки», в которых исход случаен и оторван от обучения.

### 6.3. Объяснение отказов

Если пользователь не может выполнить действие — **объяснить причину человеческим языком**, без технических деталей (если только это не учебная цель). Примеры причин:

- недостаточная `buyingPower`,
- инструмент не торгуется на `TEREX` или вне сессии,
- лимитная заявка вне допустимого ценового шага,
- `OrderStatus = rejected` — почему отклонена.

### 6.4. Микро-копи и подсказки

- Короткие, по делу, появляются в момент действия.
- Формат: «Сейчас ты делаешь X. Это значит Y.»
- Пример: «Сейчас ты выставляешь лимитную заявку. Это значит: сделка произойдёт только если в стакане найдётся встречная по твоей цене или лучше.»

## 7. UI-конвенции

- Material 3, токены из `core/theme/tokens.dart` (`AppColors.*`). Цвета захардкоженные **только** в этих токенах. В коде использовать `Theme.of(context).colorScheme.*` или `AppColors.*` — не литералы.
- Типографика — `AppTypography` (Caveat для headlines, Nunito для текста). Не подключать новые шрифты без обсуждения.
- Радиусы карточек — 14–20, нижний бар — `kNavigationBarHeight = 72`. Использовать `context.bottomNavBarPadding` для нижнего отступа в скролл-вью.
- `SafeArea` — для верхнего контента используется внутри Scaffold; `top: false` уже выставлено в обёртке нижней навигации.
- Без эмодзи в UI и без декоративных иконок без смысла. Иконки только Material Icons (или явно обсуждённые SVG).
- Реакции на состояние ошибки/загрузки — **обязательно** обработаны (никакого «вечного» CircularProgressIndicator без таймаута/ошибки).

## 9. Контент обучения

Уроки — markdown-файлы в `assets/lessons/<sectionId>/<NN>-<slug>.md`. Формат:

```
---
id: orders
title: Что такое биржа?
description: Краткое описание.
imageUrl: assets/images/lesson1.jpg
estimatedMinutes: 4
academicDefinition: Текст академического определения.
---

Тело урока в markdown...

---quiz---
[
  {
    "question": "Текст вопроса?",
    "options": ["вариант A", "вариант B"],
    "correctIndex": 1,
    "explanation": "Почему B верный."
  }
]
```

- `estimatedMinutes` и блок `---quiz---` — опциональные.
- Парсер — `features/learn/domain/learning_content_service.dart`.
- Прогресс — `features/learn/data/learning_progress_repository.dart` (SharedPreferences, без блокировок).
- Тест НЕ блокирует прохождение — только самопроверка.

## 10. Git и коммиты

Стиль коммитов — короткий префикс по типу + описание:

- `feat(<feature>): ...` — новая функциональность
- `fix(<feature>): ...` — баг-фикс
- `refactor(...): ...`
- `chore: ...`
- `docs: ...`

Пример: `feat(learn): прогресс, дорожка уроков и мини-тесты`.

В коммите — что и зачем, на русском. Не пушить генерёнку (`*.g.dart` / `*.freezed.dart`) отдельно от исходников, которые её породили.

Force-push, удаление веток, history rewrite — только с явного подтверждения пользователя.

## 11. Безопасность и данные

- Токены и секреты — только через `Storage` (secure storage на мобилке).
- Чувствительные данные (PII, токены) **не логировать**, даже при `enableLogging=true`.
- В URL-параметры не класть пользовательские данные.
- Никаких сторонних аналитик / трекеров без явного согласования.

---

## 12. Чего ассистент НЕ делает

- Не выдаёт персональных инвестиционных или финансовых советов.
- Не обещает доходность.
- Не упрощает риск (включая игровую обёртку — см. §6.2).
- Игровые механики использует осознанно (см. §6.1), без размывания природы рынка.
- Не обходит ограничения торговой логики «потому что учебная среда» — наоборот, объясняет правила.
- Не ссылается на функции, которых нет в приложении (маржинальная торговля, опционы, фьючерсы — пока не реализованы).
- Не молча применяет костыли — см. §3.1, §3.6.
- Не делает решения, которые ломают одну из платформ — см. §3.2.
